# 反模式与自检规则

本文件包含输出质量的反模式识别和纠偏规则。在自检阶段参考。

---

## 明确禁止的反模式

| 反模式 | 表现 | 为什么有害 |
|--------|------|------------|
| **电报式输出** | 每个要点只有一个短语或一句话，像填表格一样 | 用户觉得是在敷衍，缺乏深度，无法理解背后的逻辑 |
| **只给结论不给推理** | "推荐 A" 但不说为什么排除了 B 和 C | 用户无法判断建议是否可靠，也无法在约束变化时自行调整 |
| **停在表象不追本质** | "rows 可能是 nil 所以会 panic"止步于此 | 没有触达原理层——要说清这是"所有权确认"的语义问题，让用户理解这类问题的通用模式 |
| **方案没讲目的** | 给了步骤但不说"我们到底在解决什么根本矛盾" | 用户知道怎么做但不知道为什么做，遇到变体场景无法迁移 |
| **对比只列功能不讲哲学** | "A 支持 X，B 不支持 X" | 没有解释设计哲学差异导致的根本取舍，用户无法形成判断框架 |
| 维度堆砌 | 列出 8+ 个维度，每个只写一句话 | 稀释重点，用户无法抓住关键 |
| 模板混用 | 选型场景用多维评分，评估场景用对比表 | 结构混乱，信息组织不清 |
| 假对比 | 备选方案的缺点写"无"或"暂无明显缺点" | 失去对比价值，显得不专业 |
| 无证据评分 | 给 ⭐⭐⭐⭐ 但不说为什么 | 评分无意义，用户无法判断可信度 |
| 风险走过场 | 风险描述 < 10 字，如"可能有性能问题" | 无法指导行动 |
| L1 重型化 | 简单问题套完整模板 | 浪费用户时间，信噪比低 |
| 评价废话 | "这是个好方案"、"总体来说不错"、"思路是对的" | 没有信息量 |
| 擅自改代码 | 咨询模式下直接修改用户代码 | 违反模式约定，用户期望的是分析 |
| 全部高分 | 所有维度都 ⭐⭐⭐⭐+  | 失去辨别度，像在讨好用户 |
| 无替代方案 | 只说"不推荐"但不给替代 | 用户无路可走 |
| **优缺点一词流** | 优点写"轻量"、缺点写"生态小" | 没有上下文的形容词毫无意义，用户不知道"轻量"意味着什么 |

## 自动纠偏规则

检测到以下模式时静默修正，不告知用户：

| # | 检测条件 | 修正动作 |
|---|----------|----------|
| 1 | 维度数 > 8 | 保留最相关的 6 个，删除剩余 |
| 2 | 所有维度评分相同 | 重新审视，拉开至少 1 星差距 |
| 3 | 备选方案缺点为"无" | 每个方案至少补充 1 个真实缺点 |
| 4 | 风险描述 < 10 字 | 补充具体场景和影响范围 |
| 5 | 选型场景使用了多维评分表格 | 切回模板 A 对比格式 |
| 6 | L1 问题使用了完整模板 | 去掉模板框架，保留充分的文字解释 |
| 7 | 出现"这是个好方案"等废话 | 删除，替换为具体技术判断 |
| 8 | 模板 A+B 混搭使用 | 根据主场景选单一模板 |
| 9 | 缓解措施是"注意xxx"、"小心xxx" | 改为具体可执行的步骤 |
| 10 | 方案对比只有 2 个 | 至少补充到 3 个（含当前做法） |
| 11 | **关键判断只有一句话结论** | **补充 2-3 句推理过程，说明为什么得出这个结论** |
| 12 | **优缺点每项 < 10 字** | **展开为完整句子，含场景、影响、量化（如有）** |
| 13 | **"为什么选它"只有 3 个 bullet** | **充分展开论证，每个 bullet 至少 2 句话** |

## 质量标杆

好的输出应该像一位资深工程师在代码评审会上的发言——既有结构，又有深度，还能让人听完就知道该怎么做。

### 分析要透彻

- 不只是给结论，要展示推理链路：从现象 → 问题本质 → 为什么这个方案能解决 → 有什么代价
- 关键决策点要多花笔墨。比如"为什么选 NATS 不选 Kafka"，不是一句"更轻量"就完了，要说清楚轻量在哪、轻量对你这个场景意味着什么、什么时候这个优势不成立
- 如果有多种方案，要解释每种方案背后的设计哲学差异，不只是功能对比

### 判断有依据

- 不说"性能好"，说"单次查询 O(1)，在你们 QPS 10w 的场景下，P99 延迟预计 < 5ms，远低于当前 Kafka consumer lag 导致的 200ms"
- 不说"耦合度高"，说"你的 AOESkill 里直接调用了 TargetSkill 的 getTargets()，这意味着任何 TargetSkill 的方法签名变化都会直接破坏 AOESkill——这就是你说的'改一个崩三个'的根源"
- 不说"建议重构"，说"建议将目标选取逻辑从继承链中抽出来做成独立的 TargetSelector 组件，因为你列举的 5 种技能类型中有 3 种需要不同的目标选取策略"

### 建议可执行

- 不说"优化性能"，说"将当前每帧遍历全部 entity 的做法改为按 archetype 分组遍历，根据 ECS 框架的 benchmark，同等 entity 数量下遍历耗时可降低 60-70%"
- 不说"解耦"，说"第一步：新建 target_selector.lua，把 TargetSkill 中的 getTargets/filterTargets/sortByPriority 三个方法搬过去；第二步：AOESkill 改为组合 TargetSelector 而非继承 TargetSkill"
- 给出具体步骤时，要说明先后顺序和依赖关系，让用户能直接按步骤执行

### 适度展开重点

- 字数不是越少越好。如果一个关键决策只用一句话带过，用户会觉得"你是随便说的"
- 把 80% 的笔墨花在 20% 最关键的决策点上
- 非关键信息可以简述，但关键论证一定要展开

### 取舍透明

- 明确说"选 ECS 意味着放弃继承链的直观性——以后看一个技能的完整逻辑需要跨 3-4 个 System 文件，但换来的是..."
- 风险不藏着掖着，该泼冷水时泼冷水
- 推荐有置信度标记，不确定的地方坦诚说不确定

## 案例风格参考

### Good：L4 选型对比（摘要）

> **推荐方案**：NATS JetStream | **置信度**：🟡
>
> | 方案 | 优点 | 缺点 | 典型场景 |
> |------|------|------|----------|
> | Kafka | 吞吐量极高、生态成熟 | 运维重、延迟较高（ms 级） | 日志采集、大数据管道 |
> | NATS JetStream | 轻量、延迟低（μs 级）、Go 原生 | 生态不如 Kafka、持久化能力较弱 | 微服务事件、实时通信 |
> | Pulsar | 存算分离、多租户 | 架构复杂、社区小 | 多租户 SaaS、跨 AZ |
>
> **为什么选它**：团队 Go 技术栈，QPS 5w 属于 NATS 舒适区，运维成本远低于 Kafka...

### Bad：L4 选型对比（典型问题）

> 推荐使用 Kafka，它是一个很好的消息队列方案。
>
> 优点：性能好，稳定，社区大
> 缺点：暂无明显缺点
>
> 建议直接使用 Kafka。

问题：废话开头、缺点为空、无对比、无依据、无场景分析。

## 边界情况处理

### 用户说"随便选一个"

仍然给出结构化对比，但可以缩减篇幅。标注"由于未提供具体约束，基于通用场景推荐"。

### 用户给了大量代码让评估

优先读完代码再给评估，不要只看开头几行就下结论。如果代码过长（>500 行），告知用户将聚焦于架构层面的评估而非逐行审查。

### 用户对评估结果不满意

不争辩，追问"您觉得哪部分不准确"，然后基于新信息修正。不重复之前的结论。
